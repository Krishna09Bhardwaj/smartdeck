## 1. Product Overview

SmartDeck is a private, AI-powered flashcard study tool. A student uploads a PDF (textbook chapter, class notes, revision material), and Claude AI generates a high-quality deck of flashcards from it. The app then uses the SM-2 spaced repetition algorithm to schedule card reviews — surfacing weak cards more frequently and fading mastered ones. Every deck is private to the user who created it.

**Core value loop:**
1. Upload PDF → Claude generates cards
2. Study cards → Rate recall quality (0–5)
3. SM-2 schedules next review per card
4. Dashboard shows what's due today and mastery progress

## 4. Database Schema

All tables use Row Level Security (RLS). Every policy restricts reads and writes to `auth.uid() = user_id`.

```sql
-- decks: one per uploaded PDF
CREATE TABLE decks (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title        text NOT NULL,
  description  text,
  card_count   int NOT NULL DEFAULT 0,
  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now()
);

-- cards: generated by Claude, belong to a deck
CREATE TABLE cards (
  id        uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  deck_id   uuid NOT NULL REFERENCES decks(id) ON DELETE CASCADE,
  question  text NOT NULL,
  answer    text NOT NULL,
  position  int NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now()
);

-- card_reviews: SM-2 state per card per user (lazy-created on first review)
CREATE TABLE card_reviews (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  card_id          uuid NOT NULL REFERENCES cards(id) ON DELETE CASCADE,
  user_id          uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  ease_factor      float NOT NULL DEFAULT 2.5,
  interval_days    int NOT NULL DEFAULT 1,
  repetitions      int NOT NULL DEFAULT 0,
  next_review_date date NOT NULL DEFAULT CURRENT_DATE,
  last_quality     int,
  updated_at       timestamptz NOT NULL DEFAULT now(),
  UNIQUE (card_id, user_id)
);
```

**RLS policies (all tables):**
- SELECT: `auth.uid() = user_id`
- INSERT: `auth.uid() = user_id`
- UPDATE: `auth.uid() = user_id`
- DELETE: `auth.uid() = user_id`

For `cards`, policies join through `decks` to verify ownership.

## 5. SM-2 Algorithm

SM-2 tracks three values per card per user: `ease_factor` (difficulty multiplier, min 1.3, default 2.5), `interval_days` (days until next review), and `repetitions` (consecutive successful reviews).

After each card review, the student submits a quality rating from 0–5:
- **0–2:** Failed (forgot or nearly wrong)
- **3:** Correct but required significant effort
- **4:** Correct with minor hesitation
- **5:** Perfect recall

**Update logic (pure TypeScript function in `lib/sm2.ts`):**

```
if quality < 3:
  repetitions = 0
  interval = 1
else:
  if repetitions == 0: interval = 1
  elif repetitions == 1: interval = 6
  else: interval = round(interval * ease_factor)
  repetitions += 1

ease_factor = ease_factor + (0.1 - (5 - quality) * (0.08 + (5 - quality) * 0.02))
ease_factor = max(1.3, ease_factor)
next_review_date = today + interval days
```

**UI simplification:** Instead of exposing 0–5 to students, we map four buttons:
- "Forgot" → quality 1
- "Hard" → quality 3
- "Good" → quality 4
- "Easy" → quality 5

## 6. Claude Integration

**Endpoint:** `POST /api/generate`

**Flow:**
1. Client sends multipart form with PDF file + deck title
2. API route validates file type (application/pdf) and size (≤ 10MB)
3. `pdf-parse` extracts text server-side; text is truncated to 15,000 characters
4. If extracted text is < 100 characters, reject with error: "Could not extract text — this may be a scanned PDF"
5. Claude is called with a structured prompt requesting JSON output
6. Response is parsed and validated (must be an array of `{question, answer}` objects)
7. Cards are inserted into `cards` table; `decks.card_count` is updated
8. Client receives the new deck ID and redirects to deck view

**Claude prompt pattern:**
```
You are a skilled educator creating flashcards from a student's study material.

Generate exactly 15–25 high-quality flashcards from the following text. Each card must:
- Have a clear, specific question (not vague)
- Have a thorough but concise answer (2–4 sentences max)
- Cover key concepts, definitions, relationships, and any important examples
- Avoid trivial or redundant cards

Return ONLY a valid JSON array in this exact format:
[{"question": "...", "answer": "..."}, ...]

Study material:
<text>
{extracted_text}
</text>
```

**Security:** The `ANTHROPIC_API_KEY` lives only in `.env.local` and Vercel environment variables. It is never imported in any file under `app/` client components or `components/`. The `lib/claude.ts` module is imported only from `app/api/` routes.

## 7. Application Routes

### Public routes
- `GET /` — Landing page with product description and sign-in CTA
- `GET /login` — Supabase Auth login form
- `GET /signup` — Supabase Auth signup form

### Protected routes (require active session)
- `GET /dashboard` — Due-today count, deck list, quick-start CTA
- `GET /decks/new` — PDF upload flow (drag-and-drop zone + title input)
- `GET /decks/[deckId]` — Deck overview: card list, stats, study button
- `GET /decks/[deckId]/study` — Full-screen study session (SM-2 driven)
- `GET /progress` — Mastery overview across all decks

### API routes (server-side, authenticated)
- `GET /api/decks` — List user's decks
- `POST /api/decks` — Create deck record (called after generate succeeds)
- `DELETE /api/decks/[deckId]` — Delete deck + cards (cascades via FK)
- `GET /api/decks/[deckId]/cards` — Fetch cards for a deck
- `GET /api/decks/[deckId]/due` — Fetch cards due today for a deck
- `POST /api/generate` — PDF → Claude → cards pipeline
- `POST /api/reviews` — Submit SM-2 rating for a card

## 8. Study Session Flow

1. Client fetches due cards for the deck (`/api/decks/[deckId]/due`)
2. If no cards are due: show "All caught up!" message with next review date
3. Cards are shown one at a time, question side up
4. Student clicks/taps card to flip (Framer Motion rotateY animation)
5. Answer is revealed; student taps one of four rating buttons
6. `POST /api/reviews` is called; SM-2 state is upserted in `card_reviews`
7. Next card is shown; progress bar advances
8. On completion: show session summary (cards reviewed, new intervals)

## 10. Known Constraints and Risk Mitigations

| Risk | Mitigation |
|---|---|
| Scanned PDFs produce empty text | Detect < 100 chars extracted; return clear error to user |
| Large PDFs exceed token budget | Truncate extracted text to 15,000 chars before sending to Claude |
| Vercel body size limit on PDF upload | Vercel's default serverless body limit is 1MB. Override with `export const runtime = 'nodejs'` and disable default body parsing; use `request.formData()` directly for multipart. Enforce 10MB client-side check and validate server-side before parsing. |
| Supabase Auth PKCE issues on Vercel | Use `@supabase/ssr` package (not deprecated `auth-helpers`); test auth flow on Vercel preview before building other features |
| Claude returns malformed JSON | Wrap parse in try/catch; retry once with a stricter prompt; fail gracefully with user error message |
| SM-2 math bugs | `lib/sm2.ts` is a pure function — unit test it in isolation before wiring to API |
